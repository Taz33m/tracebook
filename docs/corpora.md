# Reproducible Coinbase L3 Corpora

`tracebook-corpus` turns a Coinbase Exchange L3 snapshot and feed segment into
an integrity-checked local corpus. A corpus contains sanitized source files,
canonical normalized events, deterministic golden book state, and a manifest
that binds every artifact by size and SHA-256.

The command supports six workflows:

| Command | Purpose |
| --- | --- |
| `capture` | Synchronize a public WebSocket feed with one REST L3 snapshot and build a local corpus |
| `prepare` | Build a corpus from snapshot and feed files already on disk |
| `sample` | Materialize the bundled synthetic corpus from an installed wheel |
| `verify` | Check hashes, renormalize the feed, and reproduce the golden state exactly |
| `benchmark` | Measure streaming import/replay and replay-only phases with machine metadata |
| `compare` | Compare reports for the same corpus and disclose environment/software differences |

## Market-Data Rights

Coinbase states that Exchange market data is subject to its
[Market Data Terms of Use](https://www.coinbase.com/legal/market_data). Those
terms currently restrict redistribution without written permission. Hashing
order identifiers does not change the underlying data rights.

For that reason:

- Live captures are marked `contains_market_data=true` and
  `redistribution=not_granted` in `manifest.json`.
- The default local output convention is the root `corpora/` directory, which
  is ignored by Git.
- Live capture requires the explicit `--acknowledge-market-data-terms` flag.
- The package contains only a synthetic, documentation-shaped corpus under its
  `tracebook.corpus` resources. It is not a redistributed venue capture.
- Tracebook does not decide whether a user has separate redistribution rights.

This is a product safety boundary, not legal advice.

## Install Live Capture

Offline preparation, verification, and benchmarking use the core package. The
network client is optional:

```bash
python -m pip install "tracebook-sim[capture]"
```

The extra uses the maintained `websockets.asyncio` client. Capture never
accepts Coinbase credentials and connects only to the documented public
production or sandbox endpoints.

## Live Capture

```bash
tracebook-corpus capture corpora/btcusd-local-001 \
  --product-id BTC-USD \
  --tick-size 0.01 \
  --channel level3 \
  --post-snapshot-seconds 15 \
  --max-messages 100000 \
  --acknowledge-market-data-terms
```

Use `--environment sandbox` to select Coinbase's public sandbox endpoints.

The capture sequence follows Coinbase's documented synchronization model:

1. Connect and subscribe to `full` or compact `level3`.
2. Start consuming and sanitizing frames immediately.
3. Fetch one non-aggregated REST book with `level=3` while frames continue to
   queue through the capture writer.
4. Retain frames at or before the snapshot sequence; the adapter discards them
   during deterministic playback.
5. Continue after the snapshot for the requested duration or until the message
   bound is reached.
6. Build the normalized event stream and reject gaps, crossed books, malformed
   lifecycle messages, or simulated trades.

`--max-messages` caps all retained frames, including the synchronization queue.
If that cap is exhausted before the REST snapshot completes, capture fails and
removes its staging directory instead of emitting a corpus with no verified
post-snapshot interval. Increase the cap and retry.

References:

- [Coinbase full and level3 channels](https://docs.cdp.coinbase.com/exchange/websocket-feed/channels)
- [Coinbase sequence behavior and endpoints](https://docs.cdp.coinbase.com/exchange/websocket-feed/overview)
- [Coinbase REST product book levels](https://docs.cdp.coinbase.com/api-reference/exchange-api/rest-api/products/get-product-book)
- [Coinbase sandbox endpoints](https://docs.cdp.coinbase.com/exchange/introduction/sandbox)

The REST endpoint warns against polling level 3. Tracebook makes one snapshot
request per capture and relies on the WebSocket stream afterward.

## Sanitization

Sanitization occurs before any WebSocket frame is written to disk.

- Only fields required for replay are retained.
- Fields such as `profile_id`, `user_id`, `client_oid`, and unknown additions
  are removed.
- `order_id`, `maker_order_id`, and `taker_order_id` are replaced with keyed
  128-bit BLAKE2b pseudonyms.
- The random key used for a live corpus is never written to the manifest.
- Compact `level3` schemas are rewritten when fields are removed, so each
  following array remains self-describing and replayable.
- Frames for products other than the manifest product are dropped.

Pseudonyms preserve equality within one corpus but cannot be joined across live
corpora. The synthetic fixture uses a deterministic fixture-only key so its
checked artifacts can be regenerated byte for byte.

## Prepare Recorded Input

For files that are already local:

```bash
tracebook-corpus prepare snapshot.json feed.jsonl corpora/btcusd-local-002 \
  --product-id BTC-USD \
  --tick-size 0.01 \
  --channel full \
  --source-classification market_data \
  --source-environment production
```

`market_data` is the safe default and marks redistribution as not granted. Use
`synthetic` only for data you authored and can distribute.

## Corpus Layout

```text
corpus/
  manifest.json   identity, rights, capture, sanitization, replay, and file hashes
  snapshot.json   sanitized REST L3 snapshot
  feed.jsonl      sanitized full/level3 frames in capture order
  events.jsonl    canonical normalized MarketEvent records
  golden.json     deterministic counts, event/trade digests, and complete final depth
```

`corpus_id` is a SHA-256 identity over the complete manifest, excluding only
the `corpus_id` field itself. That binds source, rights, capture, sanitization,
tool, replay, and all four checked file records. Benchmark reports additionally
record the exact serialized `manifest.json` SHA-256.

## Verify

```bash
tracebook-corpus sample /tmp/tracebook-sample-corpus
tracebook-corpus verify /tmp/tracebook-sample-corpus
```

Verification performs more than a checksum pass. It validates safe relative
paths, checks sizes and hashes, reconstructs every normalized event, reruns the
strict book replay, and compares the complete deterministic golden output.

## Benchmark And Compare

```bash
tracebook-corpus benchmark \
  /tmp/tracebook-sample-corpus \
  --warmups 2 \
  --iterations 10 \
  --output benchmark_results/corpus-baseline.json

tracebook-corpus benchmark \
  /tmp/tracebook-sample-corpus \
  --warmups 2 \
  --iterations 10 \
  --output benchmark_results/corpus-candidate.json

tracebook-corpus compare \
  benchmark_results/corpus-baseline.json \
  benchmark_results/corpus-candidate.json \
  --output benchmark_results/corpus-comparison.json
```

The report keeps raw nanosecond samples and summarizes minimum, median, nearest
rank P95, maximum, and median event throughput for:

- `stream_import_replay`: JSONL decoding, Coinbase normalization, and replay.
- `replay_only`: replay of already normalized `MarketEvent` records.

Comparisons require the same `corpus_id`. They report machine/runtime
differences separately from dependency-version differences and do not label a
change a regression when the environments differ. These are local wall-clock
measurements, not universal throughput or latency claims.

## Regenerate The Synthetic Fixture

```bash
rm -rf src/tracebook/corpus/fixtures/coinbase-btcusd-synthetic-v1
PYTHONPATH=src python -m tracebook.corpus.cli prepare \
  examples/data/coinbase_btcusd_l3_snapshot.json \
  examples/data/coinbase_btcusd_full.jsonl \
  src/tracebook/corpus/fixtures/coinbase-btcusd-synthetic-v1 \
  --product-id BTC-USD \
  --tick-size 0.01 \
  --channel full \
  --source-classification synthetic \
  --source-environment synthetic \
  --created-at 2026-07-10T12:00:00Z
```

Run `tracebook-corpus verify` afterward. A changed corpus ID requires an
intentional review of every generated artifact.

## Python API

```python
from tracebook.corpus import (
    benchmark_coinbase_corpus,
    prepare_coinbase_corpus,
    verify_coinbase_corpus,
)

manifest = prepare_coinbase_corpus(
    "snapshot.json",
    "feed.jsonl",
    "corpora/local-session",
    product_id="BTC-USD",
    tick_size="0.01",
)
verification = verify_coinbase_corpus("corpora/local-session")
report = benchmark_coinbase_corpus("corpora/local-session")
```

Async applications can await `capture_coinbase_corpus_async()` directly. The
synchronous `capture_coinbase_corpus()` wrapper rejects use from an already
running event loop instead of attempting nested loop control.
